## 2. CRITICAL — `metadata.py:44` shape crash is NOT fixed

User log claimed the bug was patched with an isinstance guard. The committed
file still has no guard:

`lib/wayback_archiver/metadata.py:42-46`

```python
def extract_api_metadata(data: dict) -> dict:
    """Extract metadata from an Adidas API JSON response."""
    attrs = data.get("attribute_list", {})
    pricing = data.get("pricing_information", {})
    desc = data.get("product_description", {})
```

The observed failure was "adidas `/api/products/*` returned a **list** instead
of a dict." That fails one level higher than the agent's initial read — `data`
itself is the list. `data.get(...)` raises `AttributeError` on line 44 and the
fetch stage aborts for that slug.

Fix (at function entry):

```python
def extract_api_metadata(data) -> dict:
    if isinstance(data, list):
        # Common shape: [{…product…}] — unwrap the first element.
        data = data[0] if data and isinstance(data[0], dict) else {}
    if not isinstance(data, dict):
        return {}
    attrs = data.get("attribute_list") or {}
    if not isinstance(attrs, dict): attrs = {}
    # repeat for pricing / desc
```

Worth adding the same defensive dance to `extract_catalog_product` and the
JSON-LD extractor.

## 4. HIGH — Protocol II recursion is incomplete

`surface_parser.py:224-236` dispatches `extract_outlinks` correctly for
`atom`, `oembed`, `sitemap`, `collection`, `home`, and `parse_surface_file:287`
explicitly handles `json_api` via `parse_products_json`. Refs are upserted to
the ledger at `surface_parser.py:329-336`. **So discovery works at the ledger
level.**

The gap is at the **index stage**. `run_stage.py:264-326` (`run_index`):

1. Reads CDX files on disk,
2. Writes `index.json`,
3. Calls `_import_entities_from_products` to copy *index → ledger*,

but it never goes the other way: ledger-only entities (those discovered by a
prior fetch-stage surface parse and written to the ledger, but absent from
CDX) are never promoted back into `config.index_file`. That matches the log
observation that ~800 products "existed in the captures but never reached the
fetch queue."

Fix sketch:

- After step 3, enumerate ledger entities that lack a CDX row but have a
  `first_seen_in` surface, synthesize a Wayback URL (the closest-timestamp
  pick is already a helper in `lib/wayback_archiver/cdx.py`), and merge them
  into `products`.
- Gate behind a `--include-ledger` flag for the conservative default; make it
  `on` by default in the `resume` subcommand.

Pair this with a regression test: one atom feed + zero CDX rows should end
with non-empty `index.json`.

## 5. HIGH — `WAYBACK_SKIP_CC` env var not implemented

responses on a subdomain, remove it from the loop for this run.

## 6. HIGH — Alt-archives has no per-host circuit breaker

`lib/wayback_archiver/alt_archives.py` has per-request timeouts (15 s default,
lines 24/100) but no per-host consecutive-failure kill-switch. Net effect with
`alternative_archives.enabled: true` and a dead site: 228 × ~15 s × 2 archives
≈ 114 minutes of blocking lookups with a hit rate observed at 0/228. Config
default is falsy (`site_config.py:174`; `data.get("alternative_archives", {})`
returns `{}` → `.get("enabled")` is `None`), so the footgun only fires when
the user explicitly opts in — but should still trip a breaker in that case.

Fix: reuse the `Breaker` pattern from `lib/wayback_archiver/resilience.py`,
keyed `(archive_name, host)`, trip on three consecutive empties, resume after
a ten-minute cooldown.

## 7. MEDIUM — `match` stage assumes `products/` exists

The `FileNotFoundError` in the user log was from `run_match`
iterating over `config.products_dir`. `ensure_project_dirs()`
exists in `site_config.py` (creates `products/`, `links_dir`, etc.) but
`run_match` doesn't call it at entry. Same shape in `run_cdn_discover` and
`run_download`. Defensive fix:

```python
def run_match(config, dry_run=False, **_kw):
    config.ensure_project_dirs()  # idempotent
    ...
```

Cheaper than bespoke per-dir checks, and matches the "write-where-you-expect"
pattern already in `parse_surface_file:300`.

## 8. MEDIUM — Download match rate 10% vs standalone 90%

`run_stage.py:1155-1190` (`run_cdn_discover` merge) and
`lib/wayback_archiver/match.py` use:

1. Exact `filename.split("__")` prefix match,
2. Substring containment against known slugs.

Missing pieces the standalone `shopify_downloader.py --cdn-prefix
--wayback-only` path leverages (per the pablosupply yield analysis):

- **UUID suffix strip.** `download.py:49-56` already defines `UUID_SUFFIX`
  regex and uses it for dedup. It is *not* used in matcher normalization —
  `KW5M5001_abc123def456.jpg` and `KW5M5001_999xyz.jpg` currently index as
  different products.
- **SKU extraction.** Shopify filenames like `KW5M5001_...` or `YZ6U1055_...`
  carry the SKU as the first alphanumeric token. A two-line regex
  `^([A-Z0-9]{6,10})[_-]` grabs it; joined against a SKU→slug map (already
  present in `metadata.json`) gives a 50%+ boost on Adidas-style catalogs.
- **Fuzzy slug fallback.** `difflib.SequenceMatcher(None, canon, slug).ratio()`
  above 0.85 is a reliable last-ditch attribution for slugs that got
  canonicalized differently in each source.

Add all three into a single `resolve_image_to_slug(url, slug_set, sku_map)`
helper; expose via a `--match-strategy aggressive|strict` flag.
