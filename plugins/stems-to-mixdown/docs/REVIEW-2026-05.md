# `stems-to-mixdown` — multi-phase technical review

_Reviewer: rigorous-engineer mode. Date: 2026-05-05. Scope: every file currently in `/Users/salvatore/Downloads/stems-to-mixdown/`, plus comparison research against three external repos and adjacent open-source tooling._

The review is intentionally not flattering. The plugin has serious strengths (clear philosophy, honest error semantics, reversibility) and a handful of real defects (DAW-bounce null tests will fail because of pan-law, "session intake" doesn't actually preserve session timing, the project keeps growing horizontally). The recommendations below are sequenced for immediate impact.

---

## 1. Current-State Inventory

### 1.1 Architecture

The plugin is a **shell-out pipeline orchestrator**, not an audio engine. Python coordinates `ffmpeg` / `ffprobe` (and optionally `mediainfo`, `bwfmetaedit`, `wavinfo`). All sample-domain processing happens inside `ffmpeg`'s `filter_complex`. Five Python scripts plus an optional Pass 0 helper:

| Script | Role | LoC |
|---|---|---|
| `scripts/identify.py` | Pass 0a — fast triage; decides whether PT intake is needed | 358 |
| `scripts/import_session_info.py` | Pass 0b — parses Pro Tools "Session Info as Text" → partial manifest | 490 |
| `scripts/analyze.py` | Pass 1 + 2 — discovery, probing, sanity checks | 834 |
| `scripts/plan.py` | Pass 3 — format decision, peak measurement, plan emission | 388 |
| `scripts/mix.py` | Pass 4 — execute the plan via ffmpeg, write outputs + sidecar logs | 383 |
| `scripts/verify.py` | Pass 5 — re-probe outputs, optional null test | 199 |
| `references/*.md` | Doctrine, manifest schema, format matrix, tool cheatsheet, metadata rules | 689 |
| `SKILL.md`, `README.md`, `stems-to-mixdown-additions-review.md` | Skill entrypoint, README, completed-additions self-review | 680 |

Total ≈ 4,000 LoC. The split between scripts and references is about 1.7 : 1, which is a reasonable ratio for a doctrine-heavy plugin, but the scripts are individually large (`analyze.py` is 834 lines, half of which is BWF / iXML / MediaInfo enrichment). See §6 for the decomposition opportunity.

### 1.2 Data flow

```
folder → identify.json → [optional] PT export → stems.manifest.yaml + stems.session.yaml
                                              ↓
                                          analysis.json (Pass 1 + 2)
                                              ↓
                                          plan.json (Pass 3, includes scout-measured peak)
                                              ↓
                                          mixdowns/<project>_<group>.flac
                                          mixdowns/<project>_<group>.flac.log.md  (Pass 4)
                                              ↓
                                          verify.json (Pass 5, optional null test)
```

Every JSON / YAML artifact is positioned as a **contract**: downstream consumers don't re-derive, they read. This is good practice; it preserves provenance and lets a human or LLM swap any stage.

### 1.3 What it does reliably

- Stereo unity-sum mixdowns from a folder of well-named WAV/FLAC stems.
- Mono → stereo upmix via `pan=stereo|c0=c0|c1=c0` (subject to the pan-law issue called out below).
- Format-decision matrix: lossy-in-chain → 16/44.1 FLAC, otherwise highest-rate / smallest-depth FLAC.
- Pre-sum attenuation derived from a real measured peak (32-bit float scout pass), not a sum-of-input-peaks estimate.
- Sidecar `.log.md` per output, with input SHAs, exact ffmpeg command, filter graph, tool versions, and pre/post measurements.
- Optional dither at 16-bit: `aresample=osf=s16:dither_method=triangular_hp`.
- BWF / iXML metadata read into `analysis.json` for an LLM operator's reference (never auto-flowed into output tags).
- 24-in-32 disambiguation when `wavinfo` is installed; explicit `bit_depth_uncertain` warning when it's not.

### 1.4 What it attempts but handles weakly

- **Pro Tools intake.** `import_session_info.py` parses the "Session Info as Text" export, which contains track names, file references, and markers but **not the per-clip timeline offsets needed to reconstruct a session**. The bridge produces `classifications` only; it cannot do anything with sessions whose stems aren't already consolidated. See §3.
- **Different-length stems.** The pipeline relies on `amix=duration=longest`. This concatenates each stem at zero offset and pads tails with silence — it has no concept of a stem starting at bar 17 of a song. For "stems" exported from a DAW with different *content windows* but the same *timeline anchor*, this is fine. For raw clips out of a session, it is not. Commandment 14 ("silence carries intent") is at war with the `import_session_info.py` aspiration here; only one of them can win.
- **Mono-stems-centered sessions.** The `pan=stereo|c0=c0|c1=c0` upmix uses 0 dB pan law. **Pro Tools' default pan law is −2.5 dB**, with −3 dB common ([Avid DUC](https://duc.avid.com/showthread.php?t=322412)). This means any session with mono stems centered will mix ≈ 2.5 dB hotter through this skill than through Pro Tools. **Commandment 15 (DAW bounce is the reference) is contradicted by the default behavior whenever mono stems are involved.** This is the single most important defect in the audio path and is not currently surfaced anywhere. See §2.4.
- **Lossy + lossless rate-collision.** When `any_lossy=True`, `decide_output_format` forces 44.1 kHz target. If a 96 kHz lossless stem is in the same group, the filter graph silently downsamples it via `aresample=soxr:precision=28`. Pass 2 fires `rate_mismatch` (so the user is asked for `--force`), but the Pass 3 plan rationale doesn't call out the downsampling explicitly. Commandment 4 ("sample-rate conversion is a lossy operation") is technically respected, but the wording is too soft for the situation.
- **Idempotency check.** `mix.py` claims idempotency by reading the existing sidecar's text and looking for filename mentions (`_input_sha` returns `filename`, not the actual SHA). Renaming a file to the same name with different content would not invalidate the cache. This is a soft-failure of Commandment 13 in a recoverable way. See §6.
- **`measure_loudness` / `measure_dc_offset` parsing.** Both walk `ffmpeg` stderr line by line. Output formatting changes between FFmpeg versions are a known source of breakage; the regex is plausible for current versions but there is no version probe and no fallback to `-of json` (which `ebur128` doesn't natively support, but `astats` can be coerced into a metadata-style emit). The pattern is fragile.

### 1.5 Implicit assumptions

| Assumption | Reality |
|---|---|
| Stems are flat WAV/FLAC files | True for "DAW bounce per track" workflows; false for raw session-clip workflows |
| All stems start at sample 0 | True for consolidated stems; false for session reconstruction |
| Mono stems are meant to be at center with 0 dB pan law | False on every Pro Tools session |
| Filenames are informative | True for engineers who label; the reason `import_session_info.py` exists in the first place |
| Sample rate inside lossy files matches the metadata sample rate | True for compliant encoders; false occasionally for re-wrapped MP3/AAC |
| `ffprobe` JSON shape is stable | Mostly stable; `bits_per_raw_sample` is `N/A` for WAV by design (a known issue addressed by `wavinfo`) |
| `ebur128` Summary block contains exactly one "Peak:" line | Currently true with `peak=true`; would break if anyone changed to `peak=true\|sample` |

### 1.6 Areas of overbuild or fragility

- `analyze.py` has accreted three optional probes (`wavinfo`, `mediainfo`, `bwfmetaedit`) with associated install instructions, a documented "production_metadata is LLM-reference-only" doctrine, and a `probe_disagreement` red flag. **None of this enrichment changes the mixdown output.** Every byte of it serves the LLM operator's planning, not the audio. That is defensible (Commandment 11) but the scope is much larger than the script's stated job.
- `import_session_info.py` is 490 lines and produces classifications only. The most valuable thing in a Pro Tools session — the timeline — is not exported by "Session Info as Text" in a structured form, and the script does not attempt to reconstruct it. The work-to-value ratio is poor.
- The repo contains a 406-line review of a not-yet-applied additions document (`stems-to-mixdown-additions-review.md`). The review is thoughtful but most of its decisions are now reflected in the code (BWF MetaEdit gating, MediaInfo CLI shell-out, nested `production_metadata`). **This file is now historical; it should be either dated and moved into a `docs/decisions/` folder or deleted.** Keeping a "review of suggested additions" alongside the implemented additions is confusing for future readers.

## 2. Audio Workflow Assessment

The plugin's stated job is "reliable stereo sums from stems, with full provenance." Evaluated against that goal:

### 2.1 Core deliverables

| Deliverable | Status | Notes |
|---|---|---|
| Full mixdown | Works | Default `instrumental + acapella` covers most use cases |
| Instrumental | Works | "Everything except `vocal` classification" |
| Acapella | Works | "Vocal classification only"; refuses if there are no vocal stems |
| Stem-balanced preview mix | Partial | Manifest's `gains:` exists, but no shorthand for "soft preview balance"; user has to know dB |
| Rough reference master | **Out of scope by design** | Commandment 9 forbids LUFS targeting, but no separate "demo bounce that hits −14 LUFS-I for sharing" path exists. Many users want this; see §5 |
| Session reconstruction from unequal-length regions | **Not delivered** | `amix=duration=longest` plus zero-offset stems means session timing is lost |
| Quality-preserving exports | Works | 32-bit float intermediate, `amix=normalize=0`, dither only when reducing depth |

### 2.2 Gain staging

- **Unity sum is correct.** `amix=inputs=N:normalize=0:weights=1 1 1 ...` is the right call, the cheatsheet justifies it well, and the rejection of `sox --norm` and `sox --guard` is sound.
- **Pre-attenuation policy is correct in shape.** Measure → derive → apply uniformly, before sum. Clear, reversible, deterministic.
- **Pre-attenuation policy has one gap.** It targets −3 dBTP only when the measured peak is between −1 and 0 dBTP or above 0 dBTP. A measured peak of, say, −0.5 dBTP triggers attenuation; a peak of −1.2 dBTP does not, even though the difference is well within ebur128 measurement noise on short content. The threshold is fine in practice; document the rationale (it's currently buried in `compute_pre_attenuation`'s code, not in `format-decisions.md`).

### 2.3 Clipping prevention and true peak

- **True peak everywhere.** `ebur128=peak=true` is used for every measurement. Sample peak is never used, which is correct.
- **The `Peak:` regex is currently safe** (under `peak=true` only the True-peak section emits a `Peak:` line in the Summary block) but is **not robust to future ffmpeg changes or a settings tweak**. The pattern should match after a `True peak:` section guard, not on `Peak:` alone. Currently buggy if a future maintainer adds `peak=true|sample` for any reason.
- **"already-clipped source" is a warning, not an error.** Correct call. Sometimes the source is honestly clipped and the user knows that.

### 2.4 Mono / stereo and pan law — the real defect

`pan=stereo|c0=c0|c1=c0` puts the mono signal at full level in both L and R. Equivalent to a 0 dB pan-law center pan. Pro Tools defaults to **−2.5 dB**, Logic to **−3 dB**, REAPER user-configurable, Cubase to **−3 dB** by default. With three different pan laws in common use, no choice is "right," but **0 dB is the unusual choice** and is the one most likely to break null tests against any DAW.

Concrete example: a song with 8 mono drum stems summed into stereo, all panned center. Through Pro Tools at default −2.5 dB pan law, each stem contributes −2.5 dB to the L+R bus. Through this skill, each contributes 0 dB. The skill's output is **+2.5 dB hotter** on every mono stem. Eight mono stems summed centered → +2.5 dB louder than the PT bounce. Commandment 15's null-test residual will be in the −18 to −20 dBFS range, an unambiguous structural difference, not a dither artifact.

This is mostly a documentation problem (not warning the user) plus a default-choice problem (silently picking 0 dB). Recommended fix: **make pan law explicit and configurable via the manifest's `output.pan_law` field, default −3 dB, and surface it in the plan rationale.** See §7 P0-1.

### 2.5 Phase, DC offset, dead channels

- DC offset detection at ±0.001 is reasonable.
- Dead-channel detection (one channel of a stereo stem fully silent) catches a real category of routing mistake.
- **Phase is mentioned in Commandment 6 but not actually measured.** The doc says "Pass 5 verification optionally runs a mono-fold measurement and reports the stereo-to-mono level delta," but `verify.py` does no such thing. The optional null-test against a reference covers a different case. **Either implement the mono-fold measurement or remove the claim from Commandment 6.**

### 2.6 Sample rate and bit depth preservation

- Format-decision matrix is sound and conservatively biased (Commandment 1). Highest common rate / smallest common depth is the right policy.
- Mixed-rate stem groups get downsampled to 44.1 kHz when any lossy is present, **without an explicit warning in the plan rationale beyond "lossy input present."** The user can be surprised to see a 96 kHz stem silently resampled. Add explicit per-stem resample-decision lines to the plan output.
- `aresample=resampler=soxr:precision=28` is the right resampler choice.

### 2.7 Dither

- Triangular high-pass dither at any 16-bit reduction. Correct.
- Commandment 5 says "even reference bounces" — applied correctly in `mix.py`.
- 24-bit FLAC at 32-bit-float intermediate uses no dither — correct (24-bit dither is a religious argument; the noise floor is below any real signal).
- **One absent case: 24-bit output from a 32-bit-float intermediate via `aresample=osf=s32`.** The current code maps 24-bit to "no dither" because depth ≤ 16 is False. But `osf=s32` is the 32-bit container, and FLAC will then encode at 24 bits if the codec is told so. Verify in the actual sidecar log that the FLAC encoder is told `-bits_per_raw_sample 24` — it is currently not, which means a 24-in-32 input becomes a 32-in-32 output. This is a quiet violation of Commandment 1.

### 2.8 Silence padding and alignment

- `duration=longest` is correct for stems that share a timeline anchor.
- **No support for stems with different start offsets.** This is the biggest mismatch between the "DAW bounce" world (where stems share an anchor) and the "session" world (where they don't). The Pro Tools intake bridge implies the latter is supported; the audio path silently assumes the former. See §3.
- Silence padding to longest is correct; trimming silence is forbidden (Commandment 14). Both correct.

### 2.9 SoX vs ffmpeg vs Python libraries

- ffmpeg-only engine choice is correct and documented. SoX is mentioned in the cheatsheet for inspection only — that is fine.
- No call to Python audio libraries (`librosa`, `pydub`, `soundfile`, `pyloudnorm`). Good — they're slower, less battle-tested for ITU-R BS.1770, and carry transitive dependencies.
- One library worth considering: **`pyloudnorm`** for an offline EBU R128 reference if you ever want to validate ffmpeg's `ebur128` numbers ([pyloudnorm paper](https://csteinmetz1.github.io/pyloudnorm-eval/paper/pyloudnorm_preprint.pdf)). Not for production use; for a single test that ffmpeg is reporting honestly. Defer.

### 2.10 "Technical rough mix" vs "release-quality master"

The skill produces a **technical rough mix** (clean unity sum, conservative format choice, no creative processing). It is not a master. The README and SKILL.md are clear about this in Commandment 9 ("Loudness normalization is not mastering"). However, **the user-facing language doesn't yet distinguish these tiers in the plan output.** A user who passes the skill a folder of stems and gets `mixdowns/track_instrumental.flac` may assume that file is a deliverable. The plan should include an explicit "this is a unity-sum technical mix; for a release-grade master, send this output to a mastering pass" disclaimer block. See §7 P1-2.

## 3. Pro Tools / Session Parsing Assessment

### 3.1 What is realistically possible without Pro Tools

Five tiers, ranked from most achievable to least:

1. **Read loose audio files in a session folder.** Trivial. Already done.
2. **Infer stem groups from filenames.** Already done by `analyze.py`'s regex classifier.
3. **Reconstruct timelines from exported AAF / OMF / XML / ADM.** Possible but expensive. AAF is the de-facto interchange format; `pyaaf2` is the canonical Python parser ([repo](https://github.com/markreidvfx/pyaaf2)), and `OpenTimelineIO` wraps it via `otio-aaf-adapter` ([repo](https://github.com/OpenTimelineIO/otio-aaf-adapter)). AAF can carry embedded essence, which crosses the "never write source files" line — needs careful handling. Recommendation: defer; treat as a v2 feature.
4. **Read Pro Tools cache or project/session support files.** Not useful. The "Session Info as Text" export is the user-facing structured artifact; the on-disk caches are not designed to be read out of band.
5. **Parse native `.ptx` directly.** **Possible but unwise.** `zamaudio/ptformat` ([repo](https://github.com/zamaudio/ptformat)) is the only real open-source `.ptx` parser; it is C/C++ only, requires runtime decryption (`ptunxor`), and has chronic compatibility regressions across Pro Tools major versions ("Fix regression with PT5 from PT11 introduction" is a representative commit message). Maintenance burden is high; the format is undocumented and changes between releases. **Do not pursue native `.ptx` parsing.**

### 3.2 What the current `import_session_info.py` actually achieves

The bridge takes a Pro Tools "Session Info as Text" export, regex-matches headers and section markers, and produces:

- `stems.session.yaml` — session metadata, track listing, file listing, raw markers/plug-ins blocks. Reference material.
- `stems.manifest.yaml` (partial) — only `classifications` populated, derived from track names.

What the bridge **does not** produce, and what the export **does not** contain in directly machine-parseable form:

- Per-clip start/end timecodes (the export contains them in tabular text, but the parser does not extract them — the regex only catches "the line ends in a `.wav`" and grabs the filename).
- Per-track gain or pan settings.
- Send routing / bus structure.
- Plug-in chains.

This means the bridge **cannot reconstruct a session timeline**, even from an export that contains the data. It can only tell you "track 14 has clip `AudioTrack14_01.wav`, which is on disk, and the track is named 'Lead Vox'." That is genuinely useful for the classification problem — no objection — but it is not session intake. **Rename the script and re-pitch it.**

### 3.3 Recommendation

**Drop the framing of "Pro Tools intake."** The honest framing is "Pro Tools track-name borrowing for classification."

- Rename `import_session_info.py` → `import_pt_track_names.py`.
- Update `SKILL.md` and `README.md` to describe Pass 0b as "borrows track names from a Pro Tools text export to classify badly-named files."
- Do **not** advertise timeline reconstruction. Do **not** pursue native `.ptx`. Do **not** pursue AAF until at least one real user requests it.
- Keep the `stems.session.yaml` sidecar — it's cheap reference context, useful to an LLM operator.

### 3.4 If session reconstruction is genuinely the goal (not recommended)

If you eventually want to reconstruct edits-from-clips:

1. Ask the user to **consolidate stems in Pro Tools first** (Edit → Consolidate). This is one click in PT and produces flat, equal-length, anchor-aligned WAVs. The current pipeline then works perfectly.
2. If consolidation isn't possible, ask the user to **export AAF**, then use `otio-aaf-adapter` to read clip start times, durations, and source-file references. Implement a separate `reconstruct.py` that emits a per-stem `(start_offset_sec, source_path)` manifest, then extend `mix.py`'s filter graph to use `adelay` per stem. **This is a non-trivial piece of work and a different audience than the current "give me a clean acapella from this folder" use case.**

The right move is to make consolidation the documented prerequisite and keep the core skill focused.

## 4. Existing Tools and Reusable Ideas

### 4.1 The three named repos

#### `hed0rah/music-teacher` — knowledge skill, not a tool

[Repo](https://github.com/hed0rah/music-teacher). 13 reference markdown files covering composition philosophy, harmony, counterpoint, orchestration, music history, mixing fundamentals, and audio analysis. 0 stars. 3 commits. Default branch is `master`.

- **What it does:** Loads Claude with a comprehensive music-theory and audio-fundamentals reference library. `MUSIC_SKILL.md` is the entry point; specific references (`MIXING_SKILL.md`, `REFERENCE_AUDIO_ANALYSIS.md`, etc.) load on demand.
- **Overlap with stems-to-mixdown:** Conceptual only. No audio processing, no scripts, no shell-outs. Pure knowledge.
- **Reusable ideas:** The "philosophy file plus reference files" structure is similar to `SKILL.md` plus `references/`. The depth of `REFERENCE_AUDIO_ANALYSIS.md` (LUFS targets per platform, dithering, FFT analysis) overlaps with `commandments.md`. **Worth borrowing nothing directly but worth reading once for cross-pollination.**
- **Production-useful:** No — it's educational reference material, not a tool. Inspirational at most.
- **Verdict:** **Ignore.** Different category of artifact.

#### `bitwize-music-studio/claude-ai-music-skills` — large, opinionated, Suno-focused

[Repo](https://github.com/bitwize-music-studio/claude-ai-music-skills). 151 stars. CC0. 52 skills, 80+ MCP tools, Python-heavy. Active development. Targets Suno-AI-generated music workflow end-to-end (concept → lyrics → Suno prompts → audio mastering → release).

- **What it does:** Full album-production pipeline for AI-generated music. Mastering targets `-14 LUFS` for streaming. Coordinates multi-model orchestration (Opus for creative, Sonnet for reasoning, Haiku for mechanical). MCP server exposes audio analysis, lyrics processing, database tools.
- **Overlap with stems-to-mixdown:** Partial. They have an audio mastering pipeline, a 7-point QC check (loudness, clipping, silence, phase, stereo width, frequency balance, dynamic range), and stem analysis. **The overlap is primarily on QC checks, where their list could inform Pass 2 additions** (currently no stereo width or frequency balance checks).
- **Reusable ideas:**
  - **MCP server architecture.** They expose tools through MCP rather than scripts. For `stems-to-mixdown`, this would mean turning `analyze.py`, `plan.py`, `mix.py` into MCP-callable tools rather than CLIs. **Worth considering for a v2 packaging story.**
  - **Multi-model routing.** Audio mastering on Opus, validation on Haiku. Probably overkill for this skill; the work is well-suited to a single model.
  - **7-point QC.** Add stereo-width measurement (M/S balance) and frequency-balance check (band-energy ratios) to Pass 2. Both are cheap with `astats` or `aspectralstats`.
- **Philosophical mismatch:** They normalize to a streaming target. `stems-to-mixdown` explicitly refuses to normalize (Commandment 9). Different audiences, different ethics. **Don't blend their mastering opinions into yours.**
- **Verdict:** **Borrow the QC checklist (selectively); ignore the mastering pipeline. Re-evaluate the MCP packaging once the core stabilizes.**

#### `sssingleton/stuarts-skills` — light advisory, no audio engine

[Repo](https://github.com/sssingleton/stuarts-skills). 0 stars. `mixing-engineer` skill: "Elite mixing & mastering advisor. EQ, compression, vocal chains, mix templates, plugin recommendations." Other skills: music-lawyer, music-marketing-genius, finance-expert, obsidian-brain, etc.

- **What it does:** Per-domain advisor skills — Claude as a knowledgeable consultant, not a tool that processes audio.
- **Overlap with stems-to-mixdown:** None on the audio path. The `mixing-engineer` skill could *complement* `stems-to-mixdown` by providing creative-decision guidance for the human ("here's how I'd approach a hip-hop vocal chain").
- **Reusable ideas:** The "skills are individually packaged plugins" pattern. **For Sal's case, this validates the choice to keep `stems-to-mixdown` as a single focused skill rather than a plugin marketplace.**
- **Verdict:** **Ignore.** Different category of artifact (advisor vs tool).

### 4.3 The strategic insight

**Source separation (demucs) is what most "I want an acapella" requests are actually asking for**, and `stems-to-mixdown` cannot do it because it requires actual stems as input. This is the most valuable adjacent capability and should be a **sibling skill** with clear handoff: `if user has stems → stems-to-mixdown`; `if user has only a mix → stems-from-mix (demucs wrapper) → stems-to-mixdown`.

Sources for separation tooling: [demucs repo](https://github.com/facebookresearch/demucs), [2026 benchmark](https://dev.to/codesugar_lin_037a57b06a4/htdemucs-vs-bs-roformer-vs-spleeter-a-2026-audio-source-separation-benchmark-2ll8), [comparison article](https://stemsplit.io/blog/spleeter-vs-demucs).

## 5. Strategic Recommendation

### 5.1 Is the plugin solving the right problem?

**Mostly yes**, with one important framing fix and one important capability gap.

**Framing fix:** drop "Pro Tools session intake" as an aspiration. The reality is "track-name borrowing." The current Pro Tools intake creates expectation/value mismatch: users will expect timeline reconstruction and won't get it.

**Capability gap:** the plugin does nothing for users who have a finished mix and want stems out of it. That is half the "I want an acapella" market. The right answer is a **sibling skill** that wraps demucs, not new code in `stems-to-mixdown`.

### 5.2 What should be first-class

1. Stem-folder triage (`identify.py`).
2. Discovery + sanity check (`analyze.py` minus the optional probe sprawl).
3. Format-decision matrix (`plan.py`).
4. Unity-sum execution (`mix.py`).
5. Re-probe + null-test verification (`verify.py`).
6. Sidecar logs.
7. Pan-law correctness (currently broken for mono-to-stereo).
8. Manifest support.

### 5.3 What should be optional / advanced

1. Production-metadata enrichment (`wavinfo` / `mediainfo` / `bwfmetaedit`). Already correctly soft-imported; reframe as opt-in instead of always-on.

## 7. Immediate Editing Plan

### 7.1 Priority 0 — defects that affect output correctness

#### P0-1: Pan-law default

**Problem.** `pan=stereo|c0=c0|c1=c0` applies 0 dB pan law. Pro Tools default is −2.5 dB; Logic / Cubase default to −3 dB. Mono stems centered will mix +2.5 to +3 dB hotter than a DAW bounce. Breaks Commandment 15 silently.

**Fix.**
- Add `output.pan_law` to manifest schema. Allowed values: `0.0`, `-2.5`, `-3.0`, `-4.5`, `-6.0`. Default: `-3.0`.
- Update `build_filter_graph` in `plan.py` to convert pan law to a gain coefficient: `coef = 10 ** (pan_law / 20)`, then `pan=stereo|c0=<coef>*c0|c1=<coef>*c0`.
- Echo the chosen pan law in plan rationale: "Pan law: −3.0 dB (Logic / Cubase default; Pro Tools default is −2.5 dB)."
- Sanity-check warn `pan_law_default_assumed` if any mono stems are present and `pan_law` is unset (uses default).

**Files:** `scripts/plan.py`, `scripts/mix.py` (sidecar log), `references/manifest-schema.md`, `references/format-decisions.md`, `references/commandments.md` (add new commandment §16: "Pan law is a choice, declare it"), `references/tool-cheatsheet.md`.

#### P0-2: 24-bit FLAC encoding from float intermediate

**Problem.** `mix.py` uses `aresample=osf=s32` for 24-bit targets but does not pass `-bits_per_raw_sample 24` to the FLAC encoder, so the output container may be 32-in-32 rather than honest 24-bit. Quietly violates Commandment 1.

**Fix.**
- For `target_format == "flac"` and `target_depth == 24`: add `-bits_per_raw_sample 24` to the encoder args.
- For `target_format in ("wav", "aiff")` and `target_depth == 24`: switch to `pcm_s24le` / `pcm_s24be` codec (already in plan.py's `codec_map`); make sure `aresample` is `osf=s32` and the codec carries the depth.
- Add a Pass 5 verify step that probes `bits_per_raw_sample` on the output and fails if it doesn't match plan.

**Files:** `scripts/mix.py`, `scripts/verify.py`.

#### P0-3: Real SHA-anchored idempotency

**Problem.** `mix.py:_input_sha` returns `filename`, not the SHA. Renaming a file to a previous filename with different content would skip remix. Soft Commandment-13 violation.

**Fix.**
- Pass `analysis.json` (or its hash table) into `mix.py` alongside `plan.json`.
- Idempotency key becomes the sorted concatenation of input SHAs + filter-graph string, hashed.
- Sidecar log records the idempotency key explicitly.

**Files:** `scripts/mix.py`. Optionally update plan.json to embed input SHAs from analysis.json so mix.py only needs the plan.

#### P0-4: Move `mixdowns/` out of the input directory

**Problem.** Default output path is `<input>/mixdowns/`, writing into the source directory. For archival source folders this is wrong; they should be read-only.

**Fix.**
- In `plan.py`, change default output dir from `directory / "mixdowns"` to `directory.parent / f"{directory.name}-mixdowns"`.
- Keep `--output-dir` override for users who actually want the in-source layout.
- Document the change in README and SKILL.md.

**Files:** `scripts/plan.py`, `README.md`, `SKILL.md`.

#### P0-5: Ebur128 Peak parsing robustness

**Problem.** Regex `line.startswith("Peak:")` will silently grab the wrong number if anyone enables `peak=true|sample` in any of the four files.

**Fix.**
- Track a `current_section` flag — only accept `Peak:` lines after a `True peak:` section header.
- Apply consistently across `analyze.py`, `mix.py`, `plan.py`, `verify.py`.

**Files:** `scripts/analyze.py:135-166`, `scripts/mix.py:52-81`, `scripts/plan.py:203-237`, `scripts/verify.py:38-58`. Best fix is to consolidate per §6.1 into `scripts/_measure.py`.
